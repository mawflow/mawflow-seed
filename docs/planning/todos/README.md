# 待办任务治理

本目录用于记录“当前业务闭环依赖它，但本轮先假定它已完成，实际暂不实现”的待办任务。

它解决的问题不是普通开发 TODO，而是多人、多设备、跨模块迭代中最容易失控的缺口：为了让当前业务流程先跑通，临时把某个接口、页面、数据、权限、外部系统或联调能力当作已经完成。只要后续模块会依赖这个假设，就必须在这里登记，不能只留在聊天记录、代码注释或某个模块的局部待办里。

## 文件职责

| 文件 | 用途 |
| --- | --- |
| `active.md` | 当前仍未完成、被依赖、被假设已完成的待办任务台账 |
| `closed.md` | 已完成、已取消、已废弃或被替代的待办任务归档 |
| `records/_template.md` | 高风险或跨模块待办的详情页模板 |

## 何时必须登记

满足任一条件时，必须新增待办任务：

- 当前流程已经依赖某个尚未真实完成的能力。
- 代码、页面、接口或测试里使用了 mock、stub、假数据、绕过校验或固定成功返回。
- 某个模块先假设另一个模块已经提供接口、状态、字段、权限、消息、任务或发布能力。
- 第三方服务、客户系统、数据导入、支付、通知、文件、报表、权限或后台配置暂未接通，但当前业务闭环先继续开发。
- 取消该能力会影响其它模块、验收口径、发布范围或交付说明。
- 完成该能力后需要与其它模块做接口、页面、数据、权限、状态流或发布联调。

只属于单个模块内部、不会被其它模块依赖、也没有“先假设已完成”的事项，可以保留在 `docs/modules/<module-key>/module.md` 的“待办”章节。只要出现跨模块依赖，必须升级到本目录，并在模块档案里只回链 TODO-ID。

## ID 与状态

TODO-ID 使用稳定格式：

```text
TODO-YYYYMMDD-<short-slug>
```

状态值：

| 状态 | 含义 |
| --- | --- |
| `assumed_done` | 当前流程已经按“完成”使用，但真实能力尚未实现 |
| `open` | 已登记，尚未开始 |
| `in_progress` | 正在补齐 |
| `ready_for_integration` | 已实现，等待受影响模块联调 |
| `completed` | 已完成并完成必要联调或给出联调结论 |
| `cancelled` | 确认不做，已处理受影响模块的假设撤销或替代方案 |
| `superseded` | 被另一个 TODO、Story、Task 或设计方案替代 |

## active 台账字段

`active.md` 中每条记录至少包含：

| 字段 | 说明 |
| --- | --- |
| TODO-ID | 稳定引用 |
| 状态 | 当前状态 |
| 类型 | `deferred_dependency` / `assumed_done` / `integration_gap` / `external_system_gap` / `data_gap` / `release_gap` |
| 当前假设 | 当前业务闭环先把什么当成已完成 |
| 暂不实现原因 | 本轮为什么先不做 |
| 主模块 | 最终负责完成的 `module_key` |
| 受影响模块 | 依赖该假设、完成或取消后需要调整的模块 |
| 影响路径 | 页面、接口、表、配置、脚本、发布覆盖或测试路径 |
| 完成后联调建议 | 完成后建议与哪些模块、接口、页面或环境联调 |
| 取消影响 | 如果取消该待办，需要撤销哪些假设或替代哪些流程 |
| 证据 | 需求、设计、任务、提交、模块档案或代码路径 |
| 负责人/来源 | 负责人、角色、Story/Task 或提出来源 |
| 最近更新 | 日期和摘要 |

## 记录流程

1. 先读取 `.maw/modules.yaml`、`docs/modules/README.md` 和本目录台账。
2. 判断该事项是模块内局部待办，还是跨模块被依赖的待办任务。
3. 如果是跨模块或被假设已完成，分配 TODO-ID 并写入 `active.md`。
4. 主模块和受影响模块的 `module.md` 只补 TODO-ID 回链，不复制完整待办事实。
5. 高风险、跨多个 app_key、涉及数据库/支付/权限/发布/客户仓同步时，创建 `records/TODO-YYYYMMDD-<short-slug>.md` 详情页。
6. 最终说明写明 `todo_task_update_status`，包括新增、完成、取消、未涉及或未更新原因。

## 完成待办

完成待办任务时，必须先读取该 TODO 的 `active.md` 记录和受影响模块档案，再执行实现或联调。

完成后至少处理：

- 移除或替换 mock、stub、假数据、绕过逻辑和“先当成完成”的说明。
- 更新主模块和受影响模块的 `module.md`、`changelog.md` 或必要设计文档。
- 把记录从 `active.md` 移到 `closed.md`，状态写 `completed`。
- 写清联调建议或联调结果，包括页面、接口、数据、权限、状态流、发布组件和验收命令。

## 取消待办

取消待办任务时，不允许只把记录删除。必须：

- 写清取消原因和替代方案。
- 列出所有受影响模块需要撤销的假设。
- 更新相关模块档案、设计文档、验收说明或发布说明。
- 把记录移到 `closed.md`，状态写 `cancelled` 或 `superseded`。
- 给出回归建议，避免旧流程继续依赖不存在的能力。

## 多设备与多人协作口径

- `active.md` 是跨模块待办事实源。模块档案只保留 TODO-ID 回链，避免多份事实漂移。
- `.maw/project-signals.yaml` 可以引用 active TODO，并为审计、巡检、大屏和 AI 前置读取提供结构化摘要；它不替代 `active.md` 的待办事实源。
- 本机差异、临时端口、个人路径、代理和一次性排障记录留在 `.local/`，不得写入待办台账。
- 待办记录中的路径使用项目根相对路径；负责人可写角色或脱敏姓名，不写私有账号、token 或本机绝对路径。
- 多人并行时，任何人如果依赖一个尚未完成的能力，应先查询 `active.md`，已有记录就更新受影响模块和最近更新，不创建重复 TODO。
- 发生口径冲突时，以当前代码和聚合 `.maw` 配置、`active.md`、模块档案、active 设计文档的顺序核对；无法确认时把冲突写入 TODO 记录的“最近更新”或详情页，不在代码里继续扩大假设。
