# FastAdmin 服务端发布经验

## 元信息

- ID：LESSON-002
- 类型：经验
- 状态：启用
- 适用场景：把已有 FastAdmin / ThinkPHP 后端发布经验迁移到其它只使用 FastAdmin 架构的后端项目，设计或审查后端发布脚本、发布流程和验收清单。

## 背景

这条经验来自 wy3d 项目一次成功的 FastAdmin 服务端发布复盘。该项目的服务端发布以 `code/server/src/` 为基础源码，按发布目标叠加 `release/server/default` 和 `release/server/<app_key>` 覆盖层，再发布到远端站点目录。一次可靠的发布不只是上传代码，还要覆盖目标识别、环境文件隔离、数据库变更、远端备份、Nginx 检查、FastAdmin / Workerman 进程重启、菜单或插件状态修正，以及上线后验收。

这份经验只面向 FastAdmin 架构的后端发布复用。迁移时保留方法，不要照抄来源项目的目标名、域名、数据库名、远端路径或客户配置。

## 范围边界

- 只覆盖 FastAdmin / ThinkPHP 后端服务发布，包括 PHP 源码、FastAdmin 插件、后端配置、后端 SQL 变更、后台菜单权限和后端长驻进程。
- 不覆盖 uni-app 客户端、小程序、设备端、设备端 H5、静态前端站点、移动端构建产物或多端联动发布。
- 不覆盖非 FastAdmin 架构的 PHP 项目发布；其它框架可以参考 dry-run、覆盖层、备份、重启和验收思想，但不能直接套用目录和命令。
- 不覆盖服务器初始化、域名备案、证书申请、云资源采购等基础设施准备工作。

## 经验结论

FastAdmin 后端发布最好先抽象成“发布目标 + 源码基础树 + 目标覆盖层 + dry-run 计划 + 远端备份替换 + 运行期重启 + 验收查询”的固定模型。每次发布前先确认目标和数据库，发布中先备份再替换，发布后必须重启后端长驻进程并做业务验收。

这条经验在模板仓库中只是参考模型。fork 项目应根据自身服务器、仓库结构、发布审批、数据库权限和回滚策略调整。

## 推荐目录设计

在采用 MAW 模板的项目中，FastAdmin 服务端发布资料建议分散到以下位置：

- `code/server/`：服务端源码和框架内业务配置权威来源。
- `release/server/default/`：所有发布目标共享的服务端发布覆盖层。
- `release/server/<app_key>/`：某个项目、客户、租户、品牌或环境特有的服务端覆盖层。
- `.maw/releases.yaml`：记录服务端发布源码、覆盖层和目标引用。
- `.maw/environments.yaml`：记录测试、预发、生产等环境入口和远端工作目录。
- `.maw/secrets*.yaml`：保存受信任私有仓库内可提交的凭证引用或真实值；外部交付前必须按项目规则脱敏。
- `ops/scripts/`：保存发布脚本、SQL 执行脚本或发布前检查脚本。
- `artifacts/release-runs/`：保存每次发布命令、目标、备份路径、SQL 文件、验收结果和遗留风险。

如果 fork 项目目录不同，可以保留“源码树、覆盖层、目标配置、脚本、发布记录”这几个职责，不必强行照搬路径。

## 可迁移发布模型

- 基础源码：以 FastAdmin / ThinkPHP 项目源码目录作为发布树来源，例如 `code/server/src/`。
- 公共覆盖层：维护所有目标都需要的补丁、静态校验文件或公共配置，例如 `release/server/default/`。
- 目标覆盖层：按 `app_key` 或发布目标保存差异配置，例如 `release/server/<app_key>/`。
- 目标配置：发布目标应显式记录 `kind`、`app_key`、域名、远端路径、SSH 引用、数据库引用、是否包含某个插件或菜单。
- 环境文件：生产 `.env` 如需随包发布，只能放在目标覆盖层，不放在公共覆盖层，也不从本地开发源码目录误带到线上。
- 运行期保留：远端上传文件、日志、缓存、后台入口、站点级 `.user.ini`、用户保存的配置等应有白名单或保留策略。

## 标准执行顺序

1. 检查工作区状态，确认本次发布包含哪些未提交或已提交改动。
2. 执行 dry-run，只输出发布计划，不改远端。
3. 核对 dry-run 的目标字段：目标名、目标类型、域名、`app_key`、数据库、远端路径、是否包含特殊插件。
4. 如有 SQL 变更，先从目标环境文件或目标配置读取数据库连接，确认数据库名属于当前目标。
5. 在远端备份受影响表或整个站点目录，再执行 SQL。若 `mysqldump` 因 tablespace 权限失败，可用 `--no-tablespaces` 重试备份。
6. 执行实际发布。
7. 发布脚本应完成打包、上传、远端解包、权限修正、Nginx 配置检查、旧版本备份保留和发布目录切换。
8. 发布后重启 FastAdmin 相关长驻进程，尤其是 Workerman、GatewayWorker、WebSocket 或队列进程。仅替换 PHP 文件不会让这些进程自动加载新代码。
9. 做站点访问、后台入口、接口、关键数据行和业务流程验收。

## 脚本能力建议

- 支持 `--dry-run`，并让 dry-run 输出足够核对的目标字段。
- 支持 `--target <name>`，不要把“客户云”“超级云”“正式服”这类口头词隐式混在一起。
- 支持 `--profile <name>`，但配置聚合结果要能在 dry-run 中看清。
- 支持 `--allow-dirty`，用于明确允许未提交改动参与发布；默认仍应提醒或阻止脏工作区。
- 发布包生成前清理 macOS 元数据，如 `._*`、`.DS_Store`、`__MACOSX`。
- 远端历史备份自动限量保留，避免备份无限增长。
- 发布脚本不要直接重传大型依赖目录，除非依赖本身就是发布变更的一部分。
- 插件菜单或后台权限规则如果随目标不同，应由发布目标决定安装、更新或清理。

## SQL 变更经验

- SQL 发布应绑定目标配置，不要人工凭记忆输入线上库名。
- SQL 执行前先备份受影响表，备份文件路径要写入发布记录。
- 脚本应校验当前目标不是其它环境，例如客户云 SQL 不能打到超级云数据库。
- SQL 文件执行后用查询验收业务关键行，而不是只看命令退出码。
- 当 SQL 与代码互相依赖时，发布记录要写清执行顺序和回滚点。

## 验收清单

- dry-run 目标、域名、数据库、远端路径与本次发布意图一致。
- 远端已有站点目录完成备份，并能定位备份路径。
- Nginx 配置测试通过。
- FastAdmin 后台入口可访问。
- Workerman、GatewayWorker、WebSocket、队列或计划任务进程已重启。
- 关键接口或页面返回正常。
- 关键数据库行、配置项或插件菜单符合预期。
- 发布记录包含命令、目标、时间、备份路径、SQL 文件、验收结果和遗留风险。

## 反例或风险

- 把某个项目的“客户云发布”误当作所有项目的通用目标，导致代码或 SQL 打到错误环境。
- 跳过 dry-run，直接执行上传或 SQL。
- 把生产 `.env` 放进公共覆盖层，导致多个目标共用同一套密钥或数据库。
- 只发布 PHP 文件，不重启 Workerman / WebSocket 进程，线上仍跑旧逻辑。
- 用源码覆盖远端运行期配置、上传文件、日志或后台入口文件。
- SQL 只执行不备份，或备份失败后仍继续上线。
- 把一次项目里的域名、路径、库名写死到通用脚本，后续复制到其它项目时留下隐性事故点。

## 来源项目字段形态

以下字段只用于理解模型，已按模板仓库用途改写为字段形态，不应直接复制到其它项目：

- 客户云目标：`customer_<app_key>` 或项目自定义目标名。
- 目标类型：`kind: customer_cloud`、`kind: production` 或项目自定义类型。
- 目标应用：`app_key: <app_key>`。
- 特殊插件开关：例如 `include_<plugin>: false` 或 `features.<plugin>: false`。
- 远端路径：`/srv/<project>/release` 或项目约定目录；如果历史项目误写为 `realse`，新项目应优先统一为 `release`。
- 其它生产目标：按项目明确命名，不要让口头词和目标名互相猜测。
- 目标差异：插件、菜单、队列、WebSocket、定时任务等差异应由目标配置显式决定。
