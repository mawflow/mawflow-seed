# Git 提交信息规范

## 元信息

- ID：LESSON-006
- 类型：经验
- 状态：启用
- 适用场景：AI 或人工在种子仓库、派生项目中提交代码、配置、文档、脚本或模板协议改动。

## 背景

本经验来自通用 Git 提交规范的种子仓库吸收。它适合作为 MAW 模板默认基线，但派生项目已有更严格约定时，以目标项目事实为准，并只做增量补充。

## 经验结论

提交信息默认使用：

```text
<type>(<scope>): <subject>
```

`scope` 在全仓文档、纯工具配置或难以归属单模块时可以省略，写成 `<type>: <subject>`。`type` 使用小写英文，`subject` 使用中文，结尾不加标点，并明确说明“做了什么”。

常用 `type`：

| 类型 | 使用场景 |
| --- | --- |
| feat | 新增功能、模块，或对现有功能做重大优化 |
| fix | 修复 bug，不新增功能 |
| docs | 仅修改文档、注释或接口说明，不修改代码逻辑 |
| style | 代码格式调整，不修改逻辑 |
| refactor | 代码重构，无新功能、无 bug 修复 |
| perf | 性能优化 |
| test | 新增或修改测试用例 |
| chore | 构建流程、工具配置或依赖调整，不影响业务代码 |
| revert | 回滚历史提交，必须说明回滚原因 |

示例：

```text
feat(user): 新增用户注册短信验证功能
fix(pay): 修复支付超时后无法重新支付的问题
docs: 更新接口文档参数说明
chore: 调整 npm 依赖版本
revert: 回滚用户短信验证功能并说明需求变更原因
```

## 错误尝试与正确方法

- 错误尝试/症状：使用 `update`、`fix`、`修改代码` 这类模糊 subject，或把多个模块、多个功能、格式化噪声混进一个提交。
- 正确方法：一个提交只做一件可独立理解和验证的事；subject 说明做了什么，复杂提交在 body 说明为什么做、怎么做、验证结果和风险。
- 触发场景：完成一段代码、配置、文档、脚本或模板协议改动，需要按项目规则提交并推送。
- 验证方式：提交前完成相关本地检查，至少保证本地编译、语法检查或核心功能验证不失败。

## 操作建议

- 先用 `git status --short` 或项目推荐封装确认本次提交范围，只暂存当前任务实际改动。
- 单模块或单组件变更优先写 `scope`，例如 `feat(server)`、`fix(auth)`、`docs(modules)`。
- 复杂提交使用多行 commit message：首行遵守格式，body 说明背景、关键取舍、验证结果和遗留风险。
- 涉及需求、缺陷或任务编号时，可在 subject 或 body 中追加关联信息，例如 `关联 REQ2026001` 或 `Fixes #BUG2026001`。
- 涉及数据库配置、核心逻辑、发布边界或敏感配置时，提交前在任务说明或收口中提示同模块开发人员关注。

## 反例或风险

- 不要只写 `update`、`fix`、`修改代码`、`调整内容`。
- 不要把无关格式化、依赖升级、业务功能和文档大改混在一个提交。
- 不要为了满足“每日提交”而提交无法运行、明显有 bug 或未完成到可验证状态的代码。
- 不要把派生项目已有强制规范覆盖掉；种子仓库规范是默认基线，目标项目事实优先。
