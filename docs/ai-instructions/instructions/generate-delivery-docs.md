# 指令：生成交付文档事实稿

## 元信息

- ID：TINST-025
- 类型：项目指令
- 状态：启用
- 维护位置：`docs/ai-instructions/instructions/generate-delivery-docs.md`
- 推荐调用：`#生成交付文档`
- 精确调用：`#T025`
- 触发词：交付文档、生成用户手册、生成概要设计、生成部署手册、生成文档事实稿
- 适用范围：根据项目内权威来源生成给人看的 Markdown 事实稿，供用户、人工或专门文档 AI 继续加工。

## 目标

当用户要求生成交付文档时，AI 应从项目内权威来源提取事实，生成结构清晰、可复核的 Markdown 事实稿。事实稿用于承接后续人工排版、正式表达、客户化措辞或专门文档 AI 加工，不是最终客户正式文档。

## 编号规则

- 本指令为模板内置指令 `TINST-025`，精确调用使用 `#T025`。
- 后续新增模板内置指令继续使用下一个可用 TINST 编号，不重排既有编号。
- 派生业务项目已有 `PINST-XXX` 编号不得因同步本指令而重排。

## 输入要求

- 必需输入：用户要生成的文档类型，或至少说明希望生成“交付文档事实稿”。
- 可选输入：用户提供的文档模板、目标读者、项目范围、模块范围、部署环境、输出目录或文件名。
- 缺失时处理：未提供模板时生成 Markdown 事实稿；未说明文档类型时，默认生成用户手册、概要设计和部署手册的事实稿大纲，并标记缺失内容。

## 输出位置规则

- 用户指定输出路径时，按用户路径输出。
- 用户提供模板时，按模板结构生成；模板中没有事实依据的位置仍标记“待补充/待人工确认”。
- 用户未指定输出路径时，默认优先直接在会话中输出 Markdown 事实稿。
- 如果用户要求落库但没有指定路径，建议写入 `docs/delivery/generated/<YYYYMMDD>-user-manual-facts.md`、`docs/delivery/generated/<YYYYMMDD>-summary-design-facts.md`、`docs/delivery/generated/<YYYYMMDD>-deployment-manual-facts.md`。
- 仅当实际创建 `docs/delivery/generated/` 目录时，才新增一个极简 `docs/delivery/generated/README.md`，说明这里保存 AI 生成的交付文档事实稿，不是最终正式交付文档。

## 基础文档类型和事实来源

三类基础交付文档及权威来源：

1. 用户手册：来源于 `.maw/modules.yaml` 和 `docs/modules/`。
2. 概要设计：来源于 `docs/design/`。
3. 部署手册：来源于 `ops/`。

其它客户文档可以基于这三类基类文档的事实派生，例如交接说明、培训材料、验收说明、运维摘要或客户问答稿。派生文档不要求强制完整；无法确认的内容必须标记“待补充/待人工确认”。

## 执行步骤

1. 判断用户要生成的文档类型：用户手册、概要设计、部署手册、三者组合，或基于三者派生的其它事实稿。
2. 如果用户提供模板，先读取模板结构，只填充能从权威来源确认的事实；模板中缺失事实的位置标记“待补充/待人工确认”。
3. 如果用户未提供模板，按 Markdown 事实稿生成，建议包含“事实来源”“适用范围”“已确认内容”“待补充/待人工确认”“不包含内容”。
4. 如果用户未指定输出路径，默认直接在会话中输出；只有用户要求落库时才写入文件，并优先使用“输出位置规则”中的建议路径。
5. 用户手册事实只从 `.maw/modules.yaml` 和 `docs/modules/` 获取；模块边界不清时按渐进式模块发现规则标记候选，不编造功能。
6. 概要设计事实只从 `docs/design/` 获取；设计文档缺失、过期或与当前代码明显冲突时，先提示或补充对应设计来源。
7. 部署手册事实只从 `ops/` 获取；涉及真实环境、密钥、生产连接串或服务器信息时只写配置位置或脱敏描述。
8. 如果生成过程中发现权威来源缺失、过期或互相冲突，优先提示用户，或在任务范围允许时补充对应来源文档，再生成事实稿。
9. 生成稿不得替代当前代码、`.maw` 聚合配置、模块档案、设计文档和运维文档。客户文档不是 AI 后续开发的唯一事实源。
10. 完成后说明生成稿基于哪些来源、哪些内容待确认、哪些内容没有生成。

## 验证方式

- 检查输出中是否列出事实来源。
- 检查缺失内容是否标记“待补充/待人工确认”。
- 检查未写入真实密钥、token、生产连接串、`.local` 本机路径或未脱敏日志。
- 检查用户手册、概要设计、部署手册分别只引用对应权威来源，或明确说明来源缺失。
- 若新增或更新了来源文档，按对应文档类型运行最小验证和索引检查。

## 禁区

- 不得编造功能、接口、部署步骤、环境参数或客户承诺。
- 不得写真实密钥、token、生产连接串、`.local` 本机路径或未脱敏日志。
- 不得把生成稿当作最终客户正式文档。
- 不得新增 `.maw/delivery-docs.yaml`。
- 不得新增客户交付文档强检查脚本。
- 不得强制创建复杂客户文档目录。
- 不得新增 `docs/delivery/customer-docs/`。
- 不得把内部 AI 控制面文档直接打包给客户。
- 不得让客户文档反过来替代当前代码、`.maw` 聚合配置、模块档案、设计文档或运维文档。

## 冲突与覆盖规则

- 用户最新明确模板和范围优先，但不得覆盖事实源和脱敏禁区。
- 与 code-only 交付冲突时，交付包边界以 code-only 规则为准；文档事实稿可以另行生成，但不默认进入 `code/` 交付物。
- 与模块档案、设计或运维文档冲突时，以当前代码和对应权威来源为准，并标记需要人工确认。
- 与中文收口指令冲突时，本指令不要求日常收口增加交付文档影响字段；只要求开发过程中有意识维护事实来源。
