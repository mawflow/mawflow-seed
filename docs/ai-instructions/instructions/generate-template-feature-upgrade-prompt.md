# 指令：生成模板新特性轻量升级提示词

## 元信息

- ID：TINST-011
- 类型：项目指令
- 状态：启用
- 维护位置：`docs/ai-instructions/instructions/generate-template-feature-upgrade-prompt.md`
- 触发词：#模版升级、模版升级、生成模板新特性升级提示词、生成轻量升级提示词、模板特性升级提示词、把这条变动生成升级提示词、把这个新特性同步给派生项目、给派生项目的升级提示词、轻量化升级模板、同步模板新特性提示词
- 适用范围：在本仓库作为模板仓库维护时，把某个新特性、规则变更或小范围模板能力整理成可复制到目标派生项目 Codex 会话中的轻量升级提示词。

## 目标

当用户希望把本模板仓库中新产生的单项或小范围新特性同步给其它派生项目时，AI 应在 `TINST-024 模板升级策略` 的边界内生成一段可直接复制到目标项目 Codex 会话中的升级提示词。该提示词用于轻量化增量升级，不负责在当前会话中执行目标项目升级。

## 输入要求

- 必需输入：要同步的新特性、变更摘要、commit、文件列表、当前 diff，或用户明确说的“这条变动”。
- 推荐输入：源模板版本、目标项目类型、必须保留的目标项目边界、期望升级范围。
- 可选输入：目标项目路径、目标项目已有规则、要跳过的文件、需要特别强调的风险。
- 缺失时处理：
  - 若用户只说“这条变动”，先用当前模板仓库的 `git status --short`、`git log -1 --oneline`、必要时 `git diff --name-only` 或 `git show --name-only --stat` 定位本次变更。
  - 若无法判断具体新特性，先让用户补充变更范围，不要编造升级内容。
  - 若未提供本机模板仓库目录，生成提示词时用当前模板仓库的 `pwd -P` 绝对路径填入新增字段“本机模板仓库目录”；目标项目会话先检查该目录是否存在，存在则优先从本机读取源模板，不存在才进入后续兜底。
  - 若未提供源模板来源，先读取 `.maw/template-source.yaml`；提示词必须包含 `Seed 来源通道`，并说明 `内部来源通道`、`public_seed` 和 `unknown_legacy` 的处理方式。“源模板本机路径”字段仍只来自用户输入或 `.local/.maw/template-source.yaml`，提示词中保留 `<源模板本机路径>` 占位，不改变该字段原有语义。
  - 若未提供源模板版本，优先使用用户指定的 branch/tag/commit；否则用当前模板仓库 `HEAD` commit。若当前工作树有未提交改动，源模板版本写“当前工作树”，并提醒目标项目会话需要能访问本机路径。

## 执行步骤

1. 先读取 `docs/ai-instructions/README.md`、`docs/ai-instructions/experience-index.md`、`.maw/upgrade-policy.yaml`、`.maw/template-source.yaml`、`docs/ai-instructions/instructions/template-upgrade-strategy.md`、本指令，以及与新特性直接相关的最小文件。
2. 判断该需求是“生成轻量升级提示词”还是“执行完整模板派生项目升级”：
   - 单项规则、小范围文档/脚本/任务包能力，生成轻量升级提示词。
   - 涉及大范围模板协议、目录结构、组件口径、模块档案、`.maw` 多文件迁移或目标项目差异较大时，建议使用 `prompts/codex/task-packs/template-feature-upgrade-codex-tasks` 完整任务包。
3. 梳理新特性：
   - 特性名称。
   - 升级目标。
   - 参考文件列表，使用项目根相对路径。
   - 需要目标项目同步的规则或字段。
   - 目标项目不得被覆盖或破坏的边界。
   - 需要移除的旧弱口径、旧字段或旧提示词。
   - 验证方式和最终说明要求。
4. 生成一段 `text` 代码块，供用户直接复制到目标项目 Codex 会话。
5. 提示词必须使用下面的骨架，并按本次特性填充内容；高风险升级先生成迁移说明或完整任务包，不强行输出轻量提示词：

```text
请在当前目标项目中，增量升级 MAW 模板仓库的“<特性名称>”能力。

本机模板仓库目录：<生成时填入当前模板仓库的 pwd -P 绝对路径；目标会话不可访问时自动兜底>
Seed 来源通道：<内部来源通道 | public_seed | unknown_legacy；目标项目按自身事实确认>
源模板本机路径：<源模板本机路径，来自用户输入或 .local/.maw/template-source.yaml>
源模板 git 地址：<按 Seed 来源通道填写；内部项目使用受控内部源或本机 overlay，外部公开项目使用 https://github.com/mawflow/mawflow-seed.git>
公开 Seed 仓：https://github.com/mawflow/mawflow-seed
源模板版本：<生成时填入当前模板仓库 HEAD commit；如需同步未提交改动则写 当前工作树>
本机模板仓库目录读取规则：如果该目录在目标 Codex 会话中存在且可读，优先作为源模板读取；如果不存在或不可访问，再按源模板读取优先级继续解析。
源模板读取优先级：用户输入 > .local/.maw/template-source.yaml > .maw/template-source.yaml > 当前仓库；外部公开项目不得读取内部私有 Seed 源。
目标项目仓库：当前 Codex 会话所在仓库

升级目标：
<用 1-3 句说明要同步的模板新特性。>

请重点参考源模板中的这些文件：
- <项目根相对路径>

执行要求：
1. 先读取目标项目现有规则，不要整文件覆盖。
2. 先识别仓库角色和 Seed 来源通道：`内部来源通道` 使用内部受控源或本机 overlay，`public_seed` 使用公开 Seed 仓，`unknown_legacy` 或证据冲突时输出证据并要求人工确认，不自动升级。
3. 先检查“本机模板仓库目录”是否存在且可读；存在时优先从该目录读取源模板文件、commit 和差异；不存在或不可访问时，不要阻塞，继续按“源模板读取优先级”解析“源模板本机路径”、`.local/.maw/template-source.yaml`、`.maw/template-source.yaml`、git 地址或当前仓库。
4. 只同步本次特性相关能力，保留目标项目已有业务规则、模块档案、app_key、发布流程、README 和私有配置。
5. 若目标项目已有同类规则，按语义合并；若发现旧弱口径或冲突口径，先改造成更保守、更保护目标项目事实的表述。
6. 本机模板仓库目录和源模板本机路径只用于本次会话读取源模板，不写入目标项目长期可提交文档；所有写入目标项目可提交文件的项目路径使用项目根相对路径。
7. 不复制源模板仓库的 Git remote、镜像目标、密钥、本机 `.local/` 配置、测试账号或业务占位值；外部公开项目不得引用内部私有 Seed 源。
8. 运行目标项目可用的最小验证。
9. 本次升级产生实际改动后，按目标项目规则提交并推送当前分支；推送项目仓库成功后，先运行目标项目的 `ops/scripts/sync-repository-mirror.sh plan` 或等价仓库级 mirror 计划命令，以 `Configured`、`Config source`、`Auto sync`、`Target enabled` 和 `Target auto sync` 判断是否继续同步镜像。不要只凭原始 `repository_mirrors.enabled=false` 判断未启用，因为目标项目可能通过 `.local` overlay 或兼容字段启用 mirror；无法 push 或无法执行 mirror 计划时写清原因、当前 commit hash 或未提交状态，以及需要人工处理的下一步。
10. 如果命中多个 app_key 且需要发布才会生效，最终说明必须按 app_key 分别给出 `#发布` 指令，用户可复制其中一条或多条选择部分发布；发布指令和默认环境必须来自目标项目 `.maw/releases.yaml` 与 `code/<app_key>/.maw.component.yaml`。收口末尾必须询问是否“确认发布全部”，用户回复“确认发布全部/确认/是”表示发布全部待发布组件。

最终说明必须采用中文、人类优先主展示，并在末尾“技术元数据”中包含：
- 变更了哪些文件
- 是否发现并移除了旧弱口径或冲突口径
- 运行了哪些验证
- 是否已提交并推送
- experience_lookup
- module_key
- module_dossier_updated
- module_dossier_reason
- updated_module_docs
- hit_code_components
- release_update_status
- release_commands
- release_confirmation_prompt
- code_delivery_status
- todo_task_update_status
- memory_update
- local_update
- upgrade_strategy_update
- seed_repository_upgrade_suggestions
```

6. 输出提示词后，简要说明该提示词适合轻量升级；如果目标项目差异较大，应改用完整内置任务包。
7. 如果本次生成提示词需要写入仓库文件，按仓库规则验证、提交并推送；如果只是聊天输出提示词，说明未产生文件变更。
8. 最终说明必须包含“目标项目使用方式”：
   - 先写清固定入口语：“复制以下提示词到目标仓库会话执行。”
   - 如果已生成 `prompts/codex/template-upgrade-prompts/<name>-prompt.md` 或同类 handoff prompt 文件，列出项目根相对路径和当前本机绝对路径；本机绝对路径只允许用于当次最终说明，不写入可提交文档正文。
   - 如果目标仓库 Codex 会话能访问当前机器上的本地提示词文件，给出可直接复制到目标仓库会话的一行：

```text
请完成 <提示词文件本机绝对路径> 任务
```

   - 如果目标仓库 Codex 会话不能访问该本机文件，说明：打开 `<提示词文件项目根相对路径>`，复制里面的 `text` 代码块整段，粘贴到目标派生项目的 Codex 会话里执行。
   - 如果只是聊天输出提示词而没有生成文件，说明直接复制当前回答里的 `text` 代码块整段到目标派生项目 Codex 会话。

## 验证方式

- 生成的提示词包含：本机模板仓库目录、源模板本机路径、源模板 git 地址、源模板版本、本机模板仓库目录读取规则、源模板读取优先级、目标项目仓库、升级目标、参考文件、执行要求和最终说明要求。
- 参考文件全部使用项目根相对路径。
- 提示词生成文件中，“本机模板仓库目录”使用当前模板仓库 `pwd -P` 的绝对路径；“源模板本机路径”继续使用 `<源模板本机路径>` 占位或用户当次输入，不改变原字段语义。目标项目升级时不得把这两个本机路径写入目标项目长期可提交文档；git 地址来自 `.maw/template-source.yaml` 或用户输入。
- 提示词强调增量合并，不覆盖目标项目 README、业务代码、真实配置、app_key、发布流程或私有规则。
- 提示词包含目标项目完成升级后的验证、提交、推送要求，并要求推送后以仓库级 mirror 计划命令的有效输出判断是否同步镜像。
- 最终说明包含提示词文件用法：能访问本机文件时用 `请完成 <提示词文件本机绝对路径> 任务`；不能访问时打开提示词文件并复制其中 `text` 代码块整段到目标项目 Codex 会话。

## 禁区

- 不得把轻量升级提示词写成“整包复制模板仓库文件”。
- 不得把本机模板仓库目录或源模板本机路径写入目标项目长期可提交文档；这些路径只允许作为复制给 Codex 的当次执行输入。
- 不得把生成的提示词文件本机绝对路径写入提示词文件、迁移说明或其它可提交文档；本机绝对路径只在最终说明中作为当次使用方式给用户。
- 不得默认使用 GitHub 镜像作为唯一源模板仓库。
- 不得复制源模板仓库的 `.git`、remote、镜像目标、客户仓库映射、真实 secrets 或 `.local/` 私有配置。
- 不得让目标项目删除已有 app_key、模块档案、发布配置或业务文档，除非用户明确要求且目标项目事实支持。

## 冲突与覆盖规则

- 用户最新明确要求优先。
- 本指令只负责生成轻量升级提示词；如果用户要求直接升级当前目标项目，按 `TINST-009` 或对应任务包执行。
- 目标项目事实优先于模板默认值；生成的提示词必须提醒目标项目 Codex 先审计、再取舍、后落地。
- 与 `TINST-024 模板升级策略` 冲突时：本指令只负责轻量提示词文本，升级资产类型和风险边界以 `TINST-024` 为准。
- 与 `TINST-009 使用内置模板任务提示词工程` 冲突时：单项低风险特性优先生成轻量提示词，大范围或高风险升级优先使用内置任务包。
