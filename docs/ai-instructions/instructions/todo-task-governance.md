# 指令：待办任务治理

## 元信息

- ID：TINST-028
- 类型：项目指令
- 状态：启用
- 维护位置：`docs/ai-instructions/instructions/todo-task-governance.md`
- 推荐调用：`#待办任务`
- 精确调用：`#T028` 或 `#T028/待办任务`
- 触发词：#待办任务、新增待办、记录待办、完成待办、取消待办、查询待办、待办影响、假设已完成、先不做但依赖、被依赖但暂不实现、完成后联调、assumed_done、deferred_dependency、integration_gap
- 适用范围：记录、查询、完成或取消“当前业务闭环依赖它，但本轮先假定它已完成，实际暂不实现”的跨模块待办任务。

## 目标

让被依赖但暂不实现的功能缺口有稳定台账、影响面、取消后果和联调建议，避免大型项目或多人协作时把临时假设遗忘在聊天记录、mock 代码、局部模块待办或个人设备上。

## 输入要求

- 必需输入：待办内容，或待办 TODO-ID。
- 推荐输入：当前假设、暂不实现原因、主模块、受影响模块、影响路径、完成后联调建议、取消影响、证据路径。
- 缺失时处理：
  - 能从 `.maw/modules.yaml`、模块档案、当前任务描述或代码路径确认的，先自行确认。
  - 无法确认主模块时，使用 `module_candidate:<名称>` 并判断是否需要走 `TINST-021 渐进式模块发现`。
  - 无法确认受影响模块时，先记录已知影响和待确认问题，不编造。
  - 如果触发词可能只是普通模块内 TODO，而不是跨模块被依赖待办，先判断是否存在“先假设已完成”或跨模块依赖；没有则保留在模块档案。

## 执行步骤

### 新增待办

1. 读取 `docs/planning/todos/README.md`、`active.md`、`closed.md`、`.maw/modules.yaml` 和 `docs/modules/README.md`。
2. 检索 `active.md` 和 `closed.md`，避免重复登记同一缺口。
3. 判断类型：
   - 只影响单个模块且没有被其它模块依赖：写入该模块 `module.md` 的“待办”章节。
   - 被其它模块依赖、当前流程先假设完成、或完成/取消会影响联调：写入 `docs/planning/todos/active.md`。
4. 分配 TODO-ID：`TODO-YYYYMMDD-<short-slug>`。
5. 在 `active.md` 填写当前假设、暂不实现原因、主模块、受影响模块、影响路径、完成后联调建议、取消影响、证据和最近更新。
6. 主模块和受影响模块的 `module.md` 只补 TODO-ID 回链和一句影响摘要，不复制完整待办事实。
7. 高风险或跨多个 app_key 的待办，复制 `docs/planning/todos/records/_template.md` 创建详情页。
8. 最终说明写 `todo_task_update_status: 已新增 <TODO-ID>`，并列出受影响模块。

### 查询影响

1. 读取 `active.md`、必要的 `closed.md`、对应详情页和相关模块档案。
2. 输出该 TODO 的当前假设、主模块、受影响模块、影响路径、完成影响、取消影响和联调建议。
3. 查询类任务默认不改文件；如发现台账明显缺字段，可建议更新或在用户同意后补齐。

### 完成待办

1. 读取该 TODO 的 `active.md` 记录、详情页和所有受影响模块档案。
2. 对照“当前假设”，检查是否需要移除 mock、stub、假数据、绕过校验、固定成功返回或临时说明。
3. 按普通开发流程完成实现、验证和必要联调；涉及 `code/` 时按 app_key 判断发布影响。
4. 更新主模块和受影响模块的 `module.md`、`changelog.md` 或必要设计/验收文档。
5. 将记录从 `active.md` 移到 `closed.md`，关闭状态写 `completed`。
6. 最终说明列出联调结果或后续联调建议，以及 `todo_task_update_status: 已完成 <TODO-ID>`。

### 取消待办

1. 读取该 TODO 的 `active.md` 记录、详情页和受影响模块档案。
2. 写清取消原因、替代方案和受影响模块需要撤销的假设。
3. 更新相关模块档案、设计、验收或发布说明。
4. 将记录从 `active.md` 移到 `closed.md`，关闭状态写 `cancelled` 或 `superseded`。
5. 最终说明列出回归建议和 `todo_task_update_status: 已取消 <TODO-ID>`。

## 验证方式

- `docs/planning/todos/active.md` 和 `closed.md` 表头完整。
- `PROJECT_COMMANDS.md` 和 `docs/ai-instructions/README.md` 已注册 `#待办任务` / `TINST-028`。
- 跨模块待办没有只写在单个模块档案里。
- 模块档案只回链 TODO-ID，不复制完整待办事实。
- 完成或取消待办后，`active.md` 与 `closed.md` 状态一致，受影响模块已更新或说明无需更新原因。
- 涉及代码、配置、发布或客户仓同步时，按项目规则运行相关验证并收口发布影响。

## 禁区

- 不把本机路径、端口、代理、token、账号密码、生产连接串或未脱敏日志写入待办台账。
- 不删除已完成或取消的待办记录；必须移入 `closed.md`。
- 不在多个模块档案中复制同一跨模块待办的完整事实。
- 不把普通模块内局部 TODO 全部升级成全局待办，只有被依赖、被假设完成或跨模块影响的缺口才进入 `docs/planning/todos/`。
- 不在完成待办时只改台账而不处理 mock、stub、假数据或绕过逻辑。
- 不在取消待办时只删记录而不说明受影响模块如何撤销假设。

## 冲突与覆盖规则

- 用户最新明确要求优先。
- 当前代码和端工程配置、聚合 `.maw` 配置、`docs/planning/todos/active.md`、模块档案、active 设计文档依次作为冲突核对来源。
- 当 `active.md` 与模块档案冲突时，以 `active.md` 的跨模块待办事实为准，并同步修正模块档案回链。
- 与发布、客户仓库同步、密钥或生产数据策略冲突时，按更保守、更高风险的规则处理。
